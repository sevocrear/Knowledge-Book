## Введение

Этот материал разбирает интервью Andrej Karpathy про code agents, AutoResearch, multi-agent loops и переход от "я пишу код руками" к "я управляю роем агентов". Главный тезис: мы входим в эпоху, где ценность инженера смещается от микро-действий (строки кода) к макро-управлению системами, целями, метриками и автономными циклами.

---

## Коротко для 5-летнего

Раньше человек сам строил дом по кирпичику. Теперь у него есть много умных роботов-строителей. Его работа — не класть каждый кирпич, а говорить, какой дом нужен, проверять, что он получается правильным, и давать роботам новые задания.

---

## О чем это видео

Видео про то, что:

- software engineering становится orchestration engineering;
- один агент -> много агентов -> лупы без человека в середине;
- ключевая оптимизация уже не "как быстро печатать", а "как увеличить полезный token throughput";
- в research выигрывают процессы, где есть проверяемая метрика и дешёвая верификация;
- будущее интерфейсов: меньше отдельных приложений, больше API + агент как универсальный UX-слой.

---

## Главные идеи

1. **Переход 80/20 -> 20/80 (и ниже) по ручному коду**  
   Инженер все меньше печатает и все больше формулирует intent, constraints и review-стандарты.

2. **Leverage через параллелизм**  
   Самый ценный навык — запускать независимые ветки работы параллельно (несколько агентов, несколько репозиториев, разные роли: планирование, реализация, ревью).

3. **AutoResearch как снятие человека с критического пути**  
   Если цель и метрика формализованы, человека лучше убрать из каждого шага цикла: он только задает objective и guardrails.

4. **Program.md как "код организации"**  
   Исследовательская организация становится набором инструкций (кто что делает, как принимаются решения), а значит ее можно оптимизировать как программу.

5. **Jagged intelligence**  
   Модели верифицируемо сильны в одних задачах и странно слабы в других. Это не "ровный AGI", а неровный профиль способностей.

## Ключевые инсайты

### 1) "Все skill issue" как рабочая модель

Когда агент не справился, часто причина не в абсолютном ceiling модели, а в:

- плохом task decomposition;
- неявных критериях "готово";
- слабых инструкциях и памяти;
- отсутствии корректного цикла проверки.

Это полезная инженерная установка, но ее нужно балансировать: иногда ограничение реальное, а не "ты плохо промптнул".

### 2) Новая единица эффективности: token throughput под контролем человека

Грубо:

$$\text{Leverage} \approx \frac{\text{полезные агентные токены}}{\text{человеческие токены и время}}$$

Смысл: лучшее использование ИИ — когда один человек запускает и поддерживает много автономных полезных процессов, а не микроменеджит один чат.

### 3) Где AutoResearch работает лучше всего

Лучший fit:

- есть объективная метрика (loss, latency, pass@tests, recall@k);
- есть быстрая/дешёвая верификация кандидата;
- пространство экспериментов большое и параллелизуемое.

Плохой fit:

- задачи с "мягкими" критериями качества;
- высокая неоднозначность intention;
- отсутствие надежной системы оценки.

### 4) Агент-first софт и "схлопывание приложений"

Если сервисы дают API и инструменты, агент может сшить их в единый язык управления. Это убирает необходимость в части узкоспециализированных UI для конечного пользователя.

### 5) Open vs Closed динамика

Тезис: полезно иметь frontier-модели впереди и open-модели чуть позади. Это дает баланс между прогрессом и доступностью, снижая системный риск монополии.

## Правила большого пальца (rules of thumb)

1. **Формулируй objective верифицируемо**  
   "Сделай лучше" -> плохо. "Уменьши p95 latency на 20% без регрессии test-suite" -> хорошо.

2. **Разбивай задачи на независимые макро-ветки**  
   Если ветки не конфликтуют, делегируй одновременно, а не последовательно.

3. **Один агент = одна роль**  
   Planner, Implementer, Reviewer, Evaluator лучше разделять.

4. **Автоматизируй оценку раньше, чем генерацию**  
   Сначала сделай надежный evaluator, потом масштабируй генерацию кандидатов.

5. **Не доверяй "ощущению прогресса" без метрик**  
   Агент может сжечь много compute на ложной ветке.

6. **Храни "память" не только в контексте чата**  
   Нужны устойчивые артефакты: policy docs, runbooks, experiment logs, decision records.

7. **Безопасность по умолчанию**  
   Для реального мира и приватных данных: минимальные права, sandbox, audit trail, approve-gates.

## Практические to-dos

### Для individual инженера

- Настроить multi-agent workflow: минимум 3 роли (plan/build/review).
- Ввести шаблон task card: цель, ограничения, критерий done, тесты.
- Вести журнал экспериментов и промпт-политик (что реально улучшает результат).
- Отдельно мерить: throughput, defect rate, rollback rate.

### Для команды

- Описать "Program.md организации": роли, handoff, quality gates.
- Автоматизировать офлайн-верификацию PR/экспериментов.
- Ввести policy для доступа агентов к секретам/инфре.
- Делать регулярный "postmortem агентных промахов" как для инцидентов.

### Для research-цикла

- Стандартизировать leaderboard-метрики и репродуцируемые run configs.
- Отделить генерацию гипотез от верификации.
- Параллелить дешевые эксперименты, а дорогие запускать только после фильтра.

## Ограничения и риски

- **Jaggedness моделей**: сильны в коде, могут проваливаться в нюансе intent.
- **Goodhart effect**: система может переоптимизировать proxy-метрику.
- **Security/privacy**: опасно давать агентам чрезмерные права.
- **False autonomy**: "кажется автономно", но скрыто требует много ручного babysitting.
- **Орг-риски**: если процесс непрозрачен, легко потерять управляемость.

---

## Прогноз на 1-3 года

1. Появится стандартный стек для long-running агентных лупов (memory, eval, scheduling, policy).
2. "Prompting" превратится в "process engineering" (дизайн циклов, а не только формулировки).
3. В инженерии вырастет ценность людей, умеющих строить проверяемые автономные контуры.
4. В исследованиях усилится автоматизированный search на малых моделях с переносом в frontier.
5. Агент-first UX постепенно заменит часть традиционных приложений в цифровых задачах.
